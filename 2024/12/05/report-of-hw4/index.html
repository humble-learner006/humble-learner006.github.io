report of hw4

Posted on 2024-12-05 In school

Report

Problem 1

Possible solution

  • brute force: find all possible route from start to end, but this will case large runtime. (find all route cause lots of time)
  • improved brute force: using Dijkstra algorithm to find the shortest distance between any two point, and only focus on the edges I have to pass, permutate them and get all possible solution of passing necessary edges. Then connect them with start and end node. using Dijkstra algorithms to calculate the minimal distance between each part’s head and tail
  • I try to using Bellman-ford algorithm, making necessary edges weight very large negative numbers so passing them will have large bonus. However I fail to deal with negative cycle case
  • Base on Bellman-ford algorithm, I try to using layered graph to solve this, but fail during I was debugging. Maybe it is possible, but due to limited time I give it up.

Idea

This problem has nothing to do with mirror. The core of this problem is finding the best route
with given start and terminal points under the constrain of passing specific edges.

As the first part is what Dijkstra do, I only focus on how to make sure it pass specific edges. Mentioned in Possible solution I give up negative number as bonus, and large number as penalty (since they always positive). After I discuss with my friends, here comes the solution.

this show the structure of this problem, I use matrix to store the shortest distance (calculated by Dijkstra) between any two node. It is very memory inefficient but reduce runtime, a n nodes problem will create a n*n matrix. I use ~ to imply getting value from the Dijkstra matrix.

here [head, tail] means the nodes of the edges you have to pass. Permutate all pairs in {}, you get all possible cases of passing necessary node. with n pairs of node, you should get 2n∗n!2^{n} * n!2n∗n! cases. (as head and tail can exchange).
I calculate all possible cases and update the minimum value. After all permutation, I get the best solution under constrain.

Advantage

  • no need to calculate distance between any two node every time I need it, it is all in dijkstra matrix. Also in this problem the graph do not change, so only need one calculation of minimal distance using dijkstra.
  • relatively clear algorithm

About debugging

I test sample 2 and get WA

all WA are smaller than right one. So I check if the calculation part is wrong.
I take 26 23 (the second WA in sample 2) to test

finally find out that for constrained edges [head, tail], the value have to be w in (u,v,w), but not the value from dijkstra matrix. This cost me lots of time.

Problem 2

Possible solution

  • update graph every day and deal with them as sub-task of this problem. In the sub-task, traverse all possible case and update the minimal weight on this case. Then update the max weight among those minimal weight in this sub-task.
  • the above is brute force, I can not come up with another

Idea

Do problem decomposition.
A garden with many path with violets (weight), and violet number changes every day. Then it is a max min problem for every day (sub-task)

  • input handle
  • deal with update of graph
  • deal with maxmin
    • find all possible path and their minimal edges
    • compare this edges with previous largest and update it.
    • return the maxmin value

Advantage

  • intuitive, the solution is doing problem decomposition and have no abstract advance trick.
